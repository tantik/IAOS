# Transformation Strategy

Status: Draft
Owner: Konstantin / Izumi IT
Purpose: Explain how Izumi IT moves from current state to the North Star.

## Strategy

Izumi IT will move from founder-led execution to a scalable AI-native product company through progressive architecture.

Progressive architecture means building only the operating system pieces that are useful now, while shaping them so they can grow later.

## Current path

1. Start with product execution for LINE Business OS.
2. Capture only the workflows and standards that reduce repeated work, risk, or decision fatigue.
3. Use GitHub as the official memory for decisions, documentation, and review history.
4. Use AI tools for planning, implementation, and review, with human responsibility before merge.
5. Add structure when the same issue appears repeatedly or when risk increases.
6. Retire or simplify IAOS documents that no longer help execution.

## What to avoid

Avoid:
- temporary throwaway processes that cannot become future standards;
- generic business theory;
- documentation that duplicates another source of truth;
- standards that nobody uses;
- AI-generated claims that are not verified.

## Add standards only when useful

New standards should be added only when they reduce repeated work, reduce risk, clarify review, improve product quality, preserve important company knowledge, or reduce decision fatigue.

## Success measure

IAOS is working if it helps Izumi IT build better product changes faster, with clearer context, fewer repeated decisions, and a better path toward future team scale.
